# Components of the Operating System

---

## Two Major Components of an OS

```css
Operating System
├── User Space
└── Kernel Space
        │
        ▼
    Hardware
```

- **User Space:** Where user applications run; provides a convenient environment for apps.
- **Kernel Space (Kernel):** Has direct access to underlying hardware; the core of the OS.

---

## User Space

- *Does NOT have direct hardware access.*
- Provides a convenient environment for user applications to run.
- The user interacts with the OS through the user space.

**Two forms of User Space interface:**

1. **GUI (Graphical User Interface)** — The visual desktop with icons, mouse clicks, windows. This is what you normally see and interact with on Windows/macOS.

2. **CLI (Command Line Interface) / Shell** — Terminal/command prompt where you type commands (e.g., `mkdir`, `ls`).

*Both GUI and CLI do the same underlying work — they just provide different interaction styles.*

**Example:** Creating a new folder:
- GUI: Right-click → New Folder
- CLI: `mkdir movies`
- Internally, both call the same OS mechanism (`mkdir` command → system call → kernel → file system)

**Third role of User Space:** It interacts with the kernel — the user space communicates with the kernel to get hardware-level work done.

---

## Kernel

*The kernel is the heart of the OS / core component.*

- Has **direct, privileged access to hardware**.
- User space does NOT access hardware directly — it requests the kernel to do so.
- The kernel receives requests from user space, interprets them, and performs the corresponding hardware operations.
- **Very first part of the OS to load on start-up.**

A **shell**, also known as a **command interpreter**, is that part of the OS that receives commands from the users and gets them executed.

**Architecture:**
```css
User Space  ←→  Kernel  ←→  Hardware
```

---

## Functions of the Kernel

**1. Process Management**
- Scheduling processes and threads on the CPUs
- Creating and deleting both user and system processes
- Suspending and resuming processes
- Providing mechanisms for process synchronization and process communication (IPC)

**2. Memory Management**
- Allocating and deallocating RAM to processes
- Tracking free memory space
- Ensuring isolation between processes (separate memory blocks)

**3. File Management**
- Creating and deleting files
- Creating and deleting directories to organize files
- Mapping files into secondary storage
- Backup support onto a stable storage media
- Maintaining the **directory structure** as a tree:

```css
/ (root)
├── lakshya/
│   ├── OS/
│   └── lakshya.txt
└── ramesh/
```
*This tree structure makes file search efficient — you traverse down the relevant branch.*

**4. I/O Management**
- Managing and controlling all I/O devices (USB drives, game controllers, etc.)
- Notifying the user when a device is connected
- Providing access methods (e.g., showing the USB in "My Computer")
- Handles **spooling**, **buffering**, and **caching**
  - **Spooling:** Queuing multiple jobs for a slow device. Example: you send 10 PDFs to the printer at once. The printer spools all 10 jobs into a storage area because the printer prints slowly — it processes them one by one from the queue.
  - **Buffering:** Pre-loading data needed soon within a single job, so it is ready when required. Example: video streaming buffers ahead so playback doesn't stall.
  - **Caching:** Stores frequently accessed data for faster repeated access. (Covered in detail in a subsequent lecture.)

## User Mode vs Kernel Mode

- The CPU operates in two modes: **User Mode** and **Kernel Mode**.
- **Kernel Mode:** Direct hardware access; privileged operations.
- **User Mode:** Limited access; where applications run.
- *Switching between the two is done via software interrupts.*

**What is a Software Interrupt?**
- A signal generated by software (not hardware) to tell the CPU: "Stop current work and handle this important task."
- Hardware interrupts are generated by hardware components (e.g., RAM failure sends an interrupt to notify the OS).

**Example flow — creating a directory:**
1. User types `mkdir movies` in the terminal → *User Mode*
2. Command is interpreted → system needs to create a directory → kernel's File Management must handle it
3. CPU switches to *Kernel Mode* via software interrupt
4. Kernel's File Management adds a new node `movies` to the directory tree on disk
5. Success → CPU switches back to *User Mode*
6. New folder is visible in GUI

## Types of Kernels

### 1. Monolithic Kernel

- All four functions (Process Mgmt, Memory Mgmt, File Mgmt, I/O Mgmt) are inside the kernel.
- Everything is centralized and tightly coupled.

```css
User Space
──────────────────
Kernel Space
  ├── Process Management
  ├── Memory Management
  ├── File Management
  └── I/O Management
──────────────────
Hardware
```

**Advantages:**
- *Fast communication* between components — all in the same space, no mode-switch overhead between subsystems (less user mode/kernel mode overheads).

**Disadvantages:**
- *Bulky kernel* — all code in one place.
- *Memory required to run is high.*
- *Less reliable* — if one component crashes, the entire kernel crashes (highly coupled).

**Examples:** Linux, Unix, MS-DOS

---

### 2. Microkernel

- Only the most critical functions (Process Management and Memory Management) remain in the kernel.
- File Management and I/O Management are moved to **User Space**.

```css
User Space
  ├── File Management
  └── I/O Management
──────────────────
Kernel Space
  ├── Process Management
  └── Memory Management
──────────────────
Hardware
```

**Advantages:**
- *Smaller kernel size*
- *More reliable and stable* — if File/I/O management crashes, the kernel keeps running
- A crash in a user-space component doesn't bring down the entire kernel

**Disadvantages:**
- *Lower performance* — frequent switching between User Mode and Kernel Mode introduces overhead.

**Why performance suffers:**
- A process running in User Mode that needs memory allocation (kernel) → switches to Kernel Mode → gets memory → switches back → then needs to write a file (file mgmt in User Mode) → switches back to User Mode → then needs memory again → switches to Kernel Mode again...
- *Constant back-and-forth User Mode ↔ Kernel Mode switching* is expensive.

**Examples:** L4 Linux, Minix OS, Symbian OS

---

## Inter-Process Communication (IPC)

*Needed in Microkernel:* File Management (User Space) and Memory Management (Kernel Space) need to communicate.

Since two processes run in isolated, separate memory spaces, they cannot directly share data. IPC provides the mechanism.

**Two methods of IPC:**

**Method 1 — Shared Memory:**
- A shared memory block is created between two processes.
- P1 writes data to the shared memory block.
- P2 reads from the same block.
- Fast, but requires synchronization to avoid race conditions.

**Method 2 — Message Passing (via Pipe):**
- A logical pipe (channel) is established between two processes.
- Messages are sent through the pipe following a defined protocol.
- Uses system calls to create and manage the pipe.

*Both methods are valid; different scenarios favor different approaches.*

---

### 3. Hybrid Kernel

- Combines the best of both Monolithic and Microkernel — combined approach.
- Speed and design of monolithic kernel.
- Modularity and stability of microkernel.
- Keeps the most performance-critical parts in kernel space (Process Mgmt, Memory Mgmt, I/O Mgmt).
- Moves only File Management to user space (or keeps selective components in user space).
- IPC also happens but with lesser overheads than pure microkernel.

```css
User Space
  └── File Management
──────────────────
Kernel Space
  ├── Process Management
  ├── Memory Management
  └── I/O Management
──────────────────
Hardware
```

**Result:**
- Less User Mode ↔ Kernel Mode switching overhead than pure Microkernel → *better performance*
- More stable than Monolithic (some components isolated) → *better reliability*
- *Best of both worlds — a balanced trade-off*

**Examples:** macOS, Windows 7/8/10 (Windows NT)

---

### 4. Nano and Exo Kernels

- More extreme minimalist designs beyond Microkernel.
- Less common in mainstream OS; primarily research-oriented.

## Summary Diagram

```css
User
 │
 ▼
User Space (GUI / CLI / Shell)
 │  ← IPC (Shared Memory or Message Passing)
 ▼
Kernel Space
 ├── Process Management (scheduling, creation, IPC)
 ├── Memory Management (allocation, deallocation, free space tracking)
 ├── File Management (tree structure, create/delete files & dirs)
 └── I/O Management (devices, spooling, buffering, caching)
 │
 ▼
Hardware (CPU, RAM, Disk, I/O Devices)
```

- The kernel is the **first OS component loaded at startup**.
- User space is where all application execution happens.
- *The only way user space accesses hardware is through the kernel.*
