# What is Segmentation

## The Problem Paging Leaves Unsolved

Paging is a powerful technique — it eliminates external fragmentation. But it has one problem rooted in how it divides the logical address space.

Paging divides the logical address space into **fixed-size pages**, without any regard for the logical structure of the program. A program, however, is composed of meaningful logical units:

- A main function
- Several helper functions
- A stack
- A heap
- Symbol tables
- etc.

These units have **different sizes** — and importantly, they form **coherent, self-contained logical blocks**.

**The specific problem:** Because paging uses a fixed page size, a single logical unit (e.g., a function) may be split across two pages. Those two pages can then land in different frames — far apart in physical RAM.

```
Function "fun" is 4KB in size.
Page size = 2KB.

fun → split into:
  Page 0 (first half of fun)  → Frame 5
  Page 1 (second half of fun) → Frame 2
```

- The CPU executes the first half of `fun` from Frame 5. When it crosses the page boundary, it must now read instructions from Frame 2 — a completely different location in physical RAM.
- This **cross-frame jump in the middle of a function** is inefficient — the CPU's memory access pattern is now non-contiguous for what is logically one continuous block of code.
- *This is not a catastrophic failure — paging still works correctly — but it is an unnecessary overhead: a single logical unit is physically fragmented across two separate RAM locations.*

---

## Segmentation: The Concept

Segmentation addresses this by dividing the logical address space according to the **programmer's and compiler's view of the program** — not by fixed sizes.

*In segmentation, each logical unit of the program becomes its own segment with its own size.*

- A function is one segment. The stack is one segment. A symbol table is one segment. Each is sized exactly to what that unit needs.
- *Segment sizes are variable — each segment is exactly as large as the logical unit it represents.*
- **The compiler decides how to divide the program into segments** — it identifies which parts of the program logically belong together and assigns each to a segment.

```
Program:
┌──────────────────┐
│ main() function  │ → Segment 0   (some size)
├──────────────────┤
│ add() function   │ → Segment 1   (different size)
├──────────────────┤
│ divide() function│ → Segment 2   (different size)
├──────────────────┤
│ Stack            │ → Segment 3   (different size)
├──────────────────┤
│ print() function │ → Segment 4   (different size)
└──────────────────┘
```

Each segment has a different size. This is the fundamental difference from paging:

- **Paging**: logical address space divided into **fixed-size** chunks → called **pages**
- **Segmentation**: logical address space divided into **variable-size** chunks → called **segments**

Each segment is allocated **contiguously** in physical RAM as a whole unit — so a function's instructions are always stored contiguously, no cross-region jumps.

---

## Segment Table

Just as paging uses a page table to map pages to frames, segmentation uses a **segment table** to map segments to their physical locations.

Each entry in the segment table stores **two values** for that segment:

| Field | Meaning |
|---|---|
| **Base** | The starting physical address of this segment in RAM |
| **Limit** | The size (length) of this segment in bytes |

*Both Base and Limit are stored per segment* — because every segment has a different size, the limit varies per entry. This is unlike paging where the limit was a single global value (the page size, the same for all).

**Example segment table:**

```
Segment Table:
┌──────┬───────┬──────┐
│  Seg │ Limit │ Base │
├──────┼───────┼──────┤
│  0   │ 1000  │ 1400 │   Seg 0: 1000 bytes, starts at physical addr 1400
│  1   │  400  │ 6300 │   Seg 1:  400 bytes, starts at physical addr 6300
│  2   │  400  │ 4300 │   Seg 2:  400 bytes, starts at physical addr 4300
│  3   │ 1100  │ 3200 │   Seg 3: 1100 bytes, starts at physical addr 3200
│  4   │ 1000  │ 4700 │   Seg 4: 1000 bytes, starts at physical addr 4700
└──────┴───────┴──────┘
```

## Address Translation in Segmentation

A logical address generated by the CPU is divided into two fields:

```
Logical Address = [ Segment Number (s) | Offset within segment (d) ]
```

- **s** — index into the segment table; identifies which segment this address belongs to.
- **d** — byte offset within that segment; how far from the start of the segment this specific byte is.

**Translation steps:**

```
CPU generates logical address (s, d)
              │
              ▼
Look up Segment Table at row s
              │
              ├─ Read Base  (physical start address of segment s)
              ├─ Read Limit (size of segment s)
              │
              ▼
Is d < Limit?
              │
        YES   │   NO
              │    └──► Exception thrown: offset out of segment bounds
              │         (accessing beyond the end of this segment — illegal)
              ▼
Physical Address = Base + d
              │
              ▼
Access physical RAM at that address
```

**Why check d < Limit?** Because each segment has a defined size. If the offset exceeds the segment's size, the access would go beyond this segment into memory that doesn't belong to it — a bounds violation.

### Concrete Example

Logical address: segment 2, offset 53 → written as `(s=2, d=53)`

From the segment table: Segment 2 → Base = 4300, Limit = 400

- Bounds check: is 53 < 400? → **YES** → proceed
- Physical address = 4300 + 53 = **4353**
- Access RAM[4353]

Another example: `(s=2, d=450)`
- Bounds check: is 450 < 400? → **NO** → exception thrown. Segment 2 is only 400 bytes; offset 450 is outside it.

## Full Worked Example

**Program structure mapped to segments:**

```
Logical Address Space of a process:
┌──────────────┐
│  Sub-routine │ Seg 0: 1000 bytes
├──────────────┤
│  Stack       │ Seg 1:  400 bytes
├──────────────┤
│  Symbol Table│ Seg 2:  400 bytes
├──────────────┤
│  Main Program│ Seg 3: 1100 bytes
├──────────────┤
│  sqrt()      │ Seg 4: 1000 bytes
└──────────────┘
```

**Segment Table:**

```
Seg 0: Base=1400, Limit=1000  → RAM[1400 to 2400)
Seg 1: Base=6300, Limit=400   → RAM[6300 to 6700)
Seg 2: Base=4300, Limit=400   → RAM[4300 to 4700)
Seg 3: Base=3200, Limit=1100  → RAM[3200 to 4300)
Seg 4: Base=4700, Limit=1000  → RAM[4700 to 5700)
```

**Physical RAM layout:**

```
┌──────────────────────────────────────────────────────────────────┐
│ OS │ ...gap... │ Seg3 │ Seg2 │ Seg4 │ ...gap... │ Seg0 │ ... │ Seg1 │
└──────────────────────────────────────────────────────────────────┘
     addr:       3200  4300  4700         1400         6300
```

Key observations:
- Each segment is stored **contiguously within itself** (e.g., all of sqrt() is in one block from 4700 to 5700).
- Different segments can be **scattered across RAM** — segmentation is non-contiguous at the inter-segment level.

**Address translation example:** access `(s=2, d=53)` — the 53rd byte of the Symbol Table.
- Segment 2: Base=4300, Limit=400
- 53 < 400 → valid
- Physical address = 4300 + 53 = **4353** → RAM[4353]

## External Fragmentation in Segmentation

*Segmentation has external fragmentation.* (Paging does not.)

**Why?** Each segment must be allocated as one contiguous block in RAM. If the available free holes in RAM are smaller than a segment that needs to be loaded, the segment cannot be placed — even if the total free memory is sufficient.

**Example:**

```
Current RAM state:
[OS][ P0-Seg0 (3KB) ][ FREE 2KB ][ P0-Seg1 (4KB) ][ FREE 2KB ][ FREE 1KB ]
```

Total free = 2 + 2 + 1 = 5KB.

New process P1 arrives with:
- Segment 0: 2KB  
- Segment 1: 3KB

- **P1 Seg 0 (2KB):** can be placed in either of the 2KB free holes → allocated ✓
- **P1 Seg 1 (3KB):** needs 3KB contiguous. Free holes are 2KB and 1KB (after placing Seg 0) — no single 3KB hole exists → **cannot allocate** ✗

Total free = 5KB, request total = 5KB, but 3KB contiguous space doesn't exist → **external fragmentation**.

*The key reason: segmentation requires each segment to fit in one contiguous physical region — unlike paging, which can split a unit across multiple frames. So holes between segments create external fragmentation, just like in contiguous allocation.*

---

## Internal Fragmentation in Segmentation

*Segmentation has NO internal fragmentation.*

- A segment is created to be exactly the size of the logical unit it represents.
- The physical block allocated for it is exactly that size.
- *No space is wasted inside a segment's allocation.*

This is in contrast to paging, where the last page of a process may be only partially filled — wasting up to (page size − 1) bytes per process.

---

## Advantages of Segmentation

- *No internal fragmentation* — every byte of the allocated physical space is used.
- *Each segment is contiguous in physical RAM* — when the CPU executes a function, all its instructions are in one block. No mid-function jumps to a different RAM location. This is more cache-friendly and efficient than paging's cross-frame splits.
- *Segment table is smaller than a page table* for the same program:
  - Example: a 100KB program, page size = 2KB → 50 pages → page table has 50 entries.
  - Same program with only 2 segments (e.g., 50KB each) → segment table has just 2 entries.
  - A smaller table means less memory needed to store it, and less overhead to look it up.
- *Logically related code stays together* — the compiler groups related functions/data into one segment, leading to better spatial locality and a more organized program structure.

---

## Disadvantages of Segmentation

- *External fragmentation is possible* — as shown in the example above. Free holes between segments may not be large enough to accommodate a new segment, even when total free memory is sufficient.

- *Variable segment sizes cause problems during swapping:*
  - Recall from the medium-term scheduler (MTS) lecture: the OS uses **swapping** to manage the degree of multiprogramming. When too many processes compete for RAM, the OS moves some segments out of RAM to **backing storage** (a region on the hard disk) temporarily — this is called **swap out**. When the process needs that segment again, it is brought back — **swap in**.
  - With paging: every page is the same size → every swap operation takes the same amount of time. Predictable, uniform.
  - With segmentation: segments have different sizes. Swapping a 5KB segment is fast; swapping a 500KB segment takes much longer. This makes swap time **unpredictable and non-uniform**.
  - In a system doing frequent swaps, this variability degrades efficiency and makes timing estimates unreliable.

---

## Internal Fragmentation in Paging (Final Clarification)

Since segmentation has no internal fragmentation but paging does, this distinction is worth making explicit:

**Example:** process size = 15KB, page size = 2KB.

```
15KB ÷ 2KB = 7 full pages (14KB of data)
             + 1 partial page: only 1KB of actual data occupies a 2KB frame

Last frame:
┌──────────────┐
│  1KB  (data) │  ← real process data
├──────────────┤
│  1KB  (FREE) │  ← internal fragmentation — wasted, unusable by others
└──────────────┘
```

- The 1KB of wasted space is locked inside the frame assigned to this process — no other process can use it.
- *In paging, this is unavoidable and must be accepted. In segmentation, it never happens.*

## Paging vs Segmentation: Full Comparison

| Property | Paging | Segmentation |
|---|---|---|
| Partition unit name | Page | Segment |
| Partition size | **Fixed** — same for all (set by hardware) | **Variable** — different per logical unit |
| Who decides the size | Hardware / processor architecture | **Compiler** (based on program structure) |
| Internal fragmentation | **Yes** — last page of each process may be partially filled | **No** — segments are exact-sized |
| External fragmentation | **No** — any free frame works, regardless of location | **Yes** — each segment needs a contiguous hole |
| Memory access within unit | **Possible cross-frame jumps** — a function may span two frames | **Always contiguous** — full segment is in one block |
| Translation table | Page table (one per process) | Segment table (one per process) |
| Table entry contains | Frame number | Base address + Limit (size) |
| Table size | **Larger** — many pages per process | **Smaller** — fewer segments per process |
| Swap time | **Uniform** — all pages the same size | **Non-uniform** — segments vary in size |

---

## Modern Systems: Both Are Used Together

In practice, modern operating systems and processors **do not use paging or segmentation in isolation**. They use a **hybrid approach** that combines both.

- The two techniques complement each other: paging handles physical allocation without external fragmentation, segmentation preserves logical program structure.
- Even page sizes are not strictly fixed on modern systems — architectures like x86-64 support multiple page sizes (4KB standard, 2MB "large pages", 1GB "huge pages") allowing a tunable balance between the two approaches.
- The concepts taught here — paging and segmentation independently — are the theoretical building blocks that real hybrid implementations are built upon.
