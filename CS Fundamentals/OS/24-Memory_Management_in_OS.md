# Memory Management in OS

## Why We Need Memory Management

- In a **multiprogramming environment**, many processes reside in RAM simultaneously.
- The goal: keep multiple processes in RAM so the CPU stays busy — if one process goes for I/O or gets context-switched (due to time-sharing / time quantum expiry), another can immediately get the CPU.
- This raises two fundamental questions:
  - How does the OS decide where each process lives in physical RAM?
  - How does the OS ensure **isolation and memory protection** between processes?

---

## The Problem: Memory Isolation

Consider three processes — P1, P2, P3 — loaded into RAM at random positions:

```
RAM
┌────────────────┐ address 0
│   OS           │  (OS always occupies the start of RAM)
├────────────────┤
│   P1           │ addresses 50–60
├────────────────┤
│   P2           │ addresses 60–70
├────────────────┤
│   (free gap)   │ addresses 70–90
├────────────────┤
│   P3           │ addresses 90–100
└────────────────┘
```

- P1 is allocated addresses 50–60 (inclusive). P2 starts immediately at 60. Their boundaries are adjacent.
- Suppose P2's programmer accidentally (or maliciously) writes code that computes `current_address + 5`. If the current address is 60, the result is 65 — which falls inside P2's own space. But if P1 does the same from its last valid address 60, `60 + 5 = 65` lands in P2's region.
- A real scenario: a variable in P1's code holds a very large value (e.g., `INT_MAX`) that gets added to an address, pushing it far out of bounds into another process's memory.
- *This is a security threat*: P1 could read or overwrite P2's data. This violates the OS principle of **isolation** — P1 should not even know that P2 exists in memory, let alone be able to touch P2's data.

**The OS must prevent this.** It cannot simply rely on programmers writing correct code.

---

## How OS Provides Isolation: Logical vs Physical Address Spaces

### The Core Idea

The OS **does not give any user process direct access to physical RAM**. Instead, it inserts a translation layer and gives each process its own **virtual (logical) address space** that starts from 0 — making each process feel like it is alone in memory.

```
P1's view:   [0 ──────── 16KB]    (Logical Address Space — virtual, imaginary)
P2's view:   [0 ──────── 16KB]    (Logical Address Space — virtual, imaginary)

Actual Physical RAM:
  P1 lives at physical addresses 128–144
  P2 lives at physical addresses 64–80
```

- Both P1 and P2 "see" addresses starting from 0 in their own private logical space.
- Because each process has its own separate addressing that starts from 0, P1 cannot generate an address that belongs to P2's logical space — they are completely separate numbering systems.
- The OS silently translates each logical address to the correct physical address behind the scenes. The user/programmer never sees or touches physical addresses.

---

## Logical Address Space

- **Generated by the CPU** as it executes process instructions.
- *It is the virtual address of instructions and data as seen by the process itself.*
- Every process has its own logical address space starting from 0 up to its maximum size.
  - Example: a 16KB process has logical addresses 0 to 16KB.
  - A 5KB process has logical addresses 0 to 5KB.
- *The set of all logical addresses generated by any program is called its **Logical Address Space** (also called **Virtual Address Space**).*
- **What you see when you print a variable's address in C++** — for example, `printf("%p", &a)` — that printed address is the **logical address**, not the real physical RAM address.
- User programs run entirely within their logical address space. They have no awareness of physical RAM layout.

---

## Physical Address Space

- *The actual address location in physical main memory (RAM)* — this hardware really exists.
- Users **can never directly access** the physical address space. The OS keeps this hidden.
- Access is always indirect: `physical address = base + logical address`
- *The set of all physical addresses that correspond to a process's logical addresses is called the **Physical Address Space** of that process.*
- Range of a process's physical address space: from `R + 0` to `R + max`, where R is the base physical address (stored in the Relocation Register) and max is the size of the process's logical address space.

---

## Memory Management Unit (MMU)

*The run-time mapping from virtual (logical) address space to physical address space is done by a hardware device called the **Memory Management Unit (MMU)**.*

- It is hardware, not software — it operates automatically on every memory access.
- The MMU sits between the CPU and RAM and performs address translation transparently.

### Two Registers the MMU Uses Per Process

| Register | What it stores |
|---|---|
| **Relocation Register** | The **base** physical address where this process starts in RAM (the physical address corresponding to logical address 0 of this process) |
| **Limit Register** | The **size** of this process's logical address space — i.e., the maximum allowed logical offset |

These two values are stored in the PCB of each process and loaded into the MMU registers when that process is scheduled.

### Address Translation Flow

```
CPU generates a Logical Address (LA)
              │
              ▼
   Is LA < value in Limit Register?
              │
       YES    │    NO
              │     └──► Trap / Exception thrown
              │          ("Addressing Error" — illegal access)
              ▼
   Physical Address = Relocation Register + LA
              │
              ▼
   Access that location in physical RAM
```

**Concrete example:**
- P2 is allocated starting at physical address 64. Limit = 16 (KB).
- Relocation Register = 64, Limit Register = 16.
- P2's programmer writes code that accesses logical address 5:
  - Check: 5 < 16 → YES → Physical address = 64 + 5 = **69** → access RAM[69]. ✓
- P2's programmer accidentally writes logical address 129:
  - Check: 129 < 16 → NO → **exception thrown**. ✗
- Even though 64 + 129 = 193 lands in another process's physical space, the check stops it before the translation even happens.

**This is how isolation and memory protection work.** The OS enforces per-process address boundaries using just two values: base and limit.

---

## The Abstraction the OS Provides

- The OS tells every process: "Your addresses start at 0. Use them freely up to your size."
- Internally (in kernel mode, invisible to the user), the OS maps logical 0 to some physical address and stores this as the Relocation Register value.
- The user program in user mode only ever sees logical addresses. The physical mapping is hidden — this is **address abstraction**.
- Example: P1's logical 0 maps to physical 128. P2's logical 0 maps to physical 64. But both P1 and P2 think they start at 0.

---

## Context Switching and the MMU Registers

- Every process has its own Relocation Register value and Limit Register value — stored in its PCB.
- When the CPU scheduler selects a new process, the **dispatcher** loads that process's correct Relocation and Limit values into the MMU registers before the process starts executing.
- This means: as soon as the CPU switches to P2, all address translations automatically use P2's base and limit — not P1's.
- *Every context switch = one register update for Relocation + one for Limit. This is all that is needed to switch address translation between processes.*

> *The OS achieves isolation not by preventing memory access but by making each process believe it is alone in memory — through the abstraction of separate logical address spaces and hardware-enforced translation.*

---

## Memory Allocation Methods in Physical RAM

Once the OS understands logical/physical address separation, the next question is: **how does the OS actually place processes in physical RAM?**

Physical memory allocation has two broad categories:

```
Memory Allocation
├── Contiguous Allocation          ← this lecture
│   ├── Fixed Partitioning
│   └── Dynamic Partitioning
└── Non-Contiguous Allocation      ← next lectures
    ├── Paging
    └── Segmentation
```

---

## Contiguous Memory Allocation

*Each process is stored in a single, unbroken (contiguous) block of physical memory.* The OS picks one region of RAM large enough for the whole process and places all of it there — no splitting, no gaps within the process.

- The process's code section, data section, stack, heap — everything is in one continuous chunk in RAM.

### Fixed Partitioning

- **RAM is pre-divided into a fixed number of partitions before any process is loaded.**
- Partitions are defined at boot/system startup — they exist regardless of whether any process is running.
- Two variants:
  - **Equal-size**: every partition is the same size (e.g., all 4MB).
  - **Unequal-size**: partitions have different predefined sizes (e.g., 4MB, 6MB, 10MB, 4MB).
- **One partition holds exactly one process** — this is a hard rule. Two processes cannot share a partition.

**Example (equal-size, 4MB partitions, 3MB processes):**

```
RAM
┌──────────────┐
│   OS         │
├──────────────┤
│ Partition 1  │ 4MB ← P1 (3MB used) + 1MB wasted internally
├──────────────┤
│ Partition 2  │ 4MB ← P2 (3MB used) + 1MB wasted internally
├──────────────┤
│ Partition 3  │ 4MB ← P3 (3MB used) + 1MB wasted internally
├──────────────┤
│ Partition 4  │ 4MB ← empty (no process assigned)
└──────────────┘
```

**Problems with Fixed Partitioning:**

- **Internal Fragmentation**: the process is smaller than its partition — the leftover space *inside* the partition cannot be used by any other process.
  - Example: 3MB process in a 4MB partition → 1MB permanently wasted per partition.
- **External Fragmentation**: each partition has leftover space (1MB each × 4 partitions = 4MB total free), but this free space is scattered — it cannot be combined to satisfy a larger contiguous request.
  - Example: a new 4MB process cannot be loaded even though 4MB total is free (it's spread as 1MB fragments across four separate partitions).
- **Limitation on process size**: no process can be larger than the largest single partition.
  - If the largest partition is 4MB, a 5MB process can never be loaded — there is no room regardless of how much total RAM is free.
- **Low degree of multiprogramming**: the maximum number of simultaneous processes equals the number of partitions. Even if processes are tiny and RAM is mostly empty inside each partition, you cannot fit more processes than partitions.

---

### Dynamic Partitioning

- *The partition size is not declared in advance — a partition is created at the moment a process is loaded, sized exactly to that process's needs.*
- No pre-division of RAM. The OS treats all free RAM as a single pool and carves out what each process needs, on demand.

**Example sequence:**

```
Initial: [OS][         all free RAM          ]

P1 (5MB) arrives  → [OS][ P1:5MB ][ free ]
P2 (2MB) arrives  → [OS][ P1:5MB ][ P2:2MB ][ free ]
P3 (3MB) arrives  → [OS][ P1:5MB ][ P2:2MB ][ P3:3MB ][ free ]
P4 (4MB) arrives  → [OS][ P1:5MB ][ P2:2MB ][ P3:3MB ][ P4:4MB ]
```

Each partition is exactly the size the process needs — no extra space inside.

**Advantages over Fixed Partitioning:**
- *No internal fragmentation* — partition size matches process size exactly. Nothing wasted inside a partition.
- No hard limit on process size — a very large process can be accommodated as long as enough contiguous free RAM exists.
- Better degree of multiprogramming — more processes can fit because no space is wasted inside partitions.

**Problem — External Fragmentation (still present):**

After P1 and P3 exit, RAM looks like this:

```
RAM
┌──────────┐
│ OS       │
├──────────┤
│ FREE 5MB │  ← gap left by P1
├──────────┤
│ P2 (2MB) │  ← still running
├──────────┤
│ FREE 3MB │  ← gap left by P3
├──────────┤
│ P4 (4MB) │  ← still running
└──────────┘
```

- Total free = 5 + 3 = 8MB.
- A new 8MB process arrives. Can it be loaded? **No.** The 8MB free space is split into two non-contiguous chunks. Contiguous allocation requires one unbroken 8MB block.
- *This is External Fragmentation: the total free memory is sufficient, but it is scattered — no single contiguous block large enough exists.*
- Dynamic partitioning eliminates internal fragmentation but does not eliminate external fragmentation.
