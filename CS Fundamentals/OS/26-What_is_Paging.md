# What is Paging

## The Problem Paging Solves

Recall from the previous lecture: in dynamic partitioning (contiguous allocation), after several processes load and exit, the RAM ends up in a fragmented state like this:

```css
RAM:       [OS][ FREE 2KB ][ P3 (8KB) ][ FREE 2KB ]
Free List: [ 2KB ] → [ 2KB ] → NULL

New request: P4 needs 3KB
Result: FAIL — no contiguous 3KB block available
```

- Total free = 4KB. Request = 3KB. But the request **cannot be satisfied** because the free space is scattered.
- The solution tried earlier was **compaction** — but compaction is expensive (time-consuming, degrades system efficiency).
- **Paging solves external fragmentation entirely, without needing compaction**, by allowing a process's memory to be placed in non-contiguous locations in RAM.

---

## Core Idea: Non-Contiguous Allocation

Instead of requiring one large contiguous block for the entire process, paging **splits the process into small, equal-size pieces** and places each piece in any available location in RAM — even if those locations are far apart from each other.

```css
RAM (fragmented, scattered free frames):
┌─────────┬─────────┬─────────┬─────────┬─────────┬─────────┬─────────┬─────────┐
│  OS     │ [busy]  │  FREE   │ [busy]  │  FREE   │  FREE   │ [busy]  │  FREE   │
└─────────┴─────────┴─────────┴─────────┴─────────┴─────────┴─────────┴─────────┘

Paging solution: place piece 1 here, piece 2 there, piece 3 there...
→ No need for contiguous free space at all
```

---

## Terminology: Pages and Frames

Two new terms are introduced — both describe the same fixed-size unit, but in different address spaces:

| Term | Definition |
|---|---|
| **Page** | A fixed-size partition of a process's **logical address space** |
| **Frame** | A fixed-size partition of **physical RAM** |

- *Page size is always equal to frame size* — this is a fundamental rule of paging.
- A page from the logical address space is loaded into a frame in physical RAM.
- **Page size is determined by the processor's architecture** — it is not chosen by the programmer or the OS. Traditionally 4KB; on modern systems multiple page sizes may be supported (e.g., 4KB, 2MB, 1GB).
- For now, treat page size as a fixed constant decided by the hardware.

*Paging is a non-contiguous memory management scheme in which the logical address space of a process is divided into fixed-size units called pages, and physical RAM is divided into same-size units called frames. Pages are loaded into available frames — not necessarily adjacent.*

### Visual Layout

```css
Logical Address Space (64KB process, 16KB pages → 4 pages):
┌──────────┐
│  Page 0  │  16KB   ← logical addresses 0   to 16KB
├──────────┤
│  Page 1  │  16KB   ← logical addresses 16KB to 32KB
├──────────┤
│  Page 2  │  16KB   ← logical addresses 32KB to 48KB
├──────────┤
│  Page 3  │  16KB   ← logical addresses 48KB to 64KB
└──────────┘

Physical RAM (128KB, divided into 16KB frames → 8 frames):
┌──────────┐  Frame 0  ← OS
├──────────┤  Frame 1  ← [another process]
├──────────┤  Frame 2  ← Page 3 of our process
├──────────┤  Frame 3  ← Page 0 of our process
├──────────┤  Frame 4  ← [another process]
├──────────┤  Frame 5  ← Page 2 of our process
├──────────┤  Frame 6  ← [another process]
└──────────┘  Frame 7  ← Page 1 of our process
```

The pages of our process are scattered across frames 2, 3, 5, 7 — none are adjacent in RAM. That is perfectly fine with paging.

## Page Table

To translate logical addresses to physical addresses, the OS maintains a **page table** for each process.

- The page table records: **which frame each page is stored in**.
- It is essentially an array indexed by page number; each entry holds the corresponding frame number.
- *Every process has its own page table* — because every process has a different mapping of its pages to physical frames.
- The page table is stored in **RAM** (as part of the OS's kernel data for that process).
- The base address of the page table is stored in the **PCB (Process Control Block)** of the process, so the OS can find it when needed.

**Example page table (for the process above):**

| Page Number (index) | Frame Number (stored value) |
|---|---|
| 0 | 3 |
| 1 | 7 |
| 2 | 5 |
| 3 | 2 |

Reading: "Page 0 of this process is stored in Frame 3 of physical RAM."

## Address Translation in Paging

A logical address generated by the CPU is divided into two fields:

```css
Logical Address = [ Page Number (p) | Offset within page (d) ]
```

- **p** — index into the page table; identifies which page this address belongs to.
- **d** — byte offset within that page; how far into the page this specific byte is.

After translation:

```css
Physical Address = [ Frame Number (f) | Offset (d) ]
```

- **f** — obtained by looking up page number p in the page table.
- **d** — *the offset is identical in both logical and physical addresses*; it is copied directly without change.

*Only the page number p is replaced by the frame number f. The offset d stays the same.*

### Bit-Level Example

**Given:**
- Logical address space = 64 bytes → needs 6 bits (2⁶ = 64)
- Page size = 16 bytes → offset needs 4 bits (2⁴ = 16) → **d = 4 bits**
- Number of pages = 64 / 16 = 4 → page number needs 2 bits (2² = 4) → **p = 2 bits**
- Physical RAM = 128 bytes → needs 7 bits; 128 / 16 = 8 frames → frame number needs 3 bits → **f = 3 bits**

**Logical address 25** in binary:

```css
25 = 0 1 1 0 0 1  (6 bits)
     [p=01] [d=1001]
      ↓
      p = 01 (binary) = 1 (decimal) → look up page 1 in page table
      page table says: page 1 → frame 7 → frame 7 in binary = 111
      ↓
Physical address = [f=111] [d=1001]  (7 bits)
                 = 1 1 1 1 0 0 1  = 121 (decimal)
```

**Verification:** Page 1 starts at logical address 16 (= 1 × 16). Offset = 25 − 16 = 9. Frame 7 starts at physical address 112 (= 7 × 16). Physical address = 112 + 9 = **121**. ✓

*The offset (d = 1001 = 9) is identical in both the logical and physical addresses. Only the leading bits changed from page number 01 (= page 1) to frame number 111 (= frame 7).*

## How Paging Eliminates External Fragmentation

Consider the previous fragmentation scenario: a 64KB process arrives but RAM has only scattered 16KB free frames, with no 64KB contiguous block.

- **Contiguous allocation:** fails — cannot find one unbroken 64KB region.
- **Paging:** split the process into four 16KB pages. Place each page into any available 16KB free frame, regardless of where those frames are in RAM.
- *No contiguous free space is required. Each page independently finds a free frame.*
- **External fragmentation is completely eliminated** — as long as enough total free frames exist, the process can always be loaded.

---

## Page Table Base Register (PTBR) and Context Switching

Since every process has its own page table stored in RAM, the system needs a way to quickly locate the current process's page table.

- A hardware register called the **Page Table Base Register (PTBR)** holds the memory address (base address) of the currently running process's page table.
- At **context switch**: the dispatcher retrieves the new process's page table address from its PCB, then loads that address into the PTBR.
- *Switching page tables at context switch requires updating only this one register — the PTBR.*
- All subsequent address translations automatically use the new process's page table.

---

## Why Paging is Slow: The Double Memory Access Problem

Paging introduces a performance cost that does not exist in simple contiguous allocation.

**The problem:** the page table itself is stored in RAM.

To access any single memory location, the CPU must make **two separate RAM accesses**:

1. **First RAM access:** read the page table (stored in RAM) to find the frame number for this page.
2. **Second RAM access:** now that the physical address is known, access the actual data at that physical address.

*Every memory reference in a paging system costs two memory accesses instead of one.* In a system with many processes doing millions of memory accesses per second, this overhead doubles the effective memory access time — a significant performance penalty.

---

## TLB: Translation Lookaside Buffer

To solve the double memory access problem, hardware provides the **Translation Lookaside Buffer (TLB)**.

- *The TLB is a hardware device* — not software, not ordinary RAM. It is a very small, very fast associative memory (hardware cache) built directly into the processor chip.
- **TLB access time << RAM access time.** (TLB is more expensive hardware than RAM — smaller in capacity but much faster.)
- The TLB stores a small number of recently used **{page number → frame number}** pairs.

### How the TLB Works

```css
CPU generates logical address (p, d)
              │
              ▼
     Look up page number p in TLB
              │
     ┌────────┴────────┐
   TLB HIT           TLB MISS
     │                   │
Frame number f          Go to page table in RAM
found instantly         Look up page p → get frame f
                        Update TLB: store {p, f} for next time
     │                   │
     └─────────┬─────────┘
               ▼
   Physical address = frame f + offset d
               │
               ▼
         Access RAM (actual data)
```

- **TLB Hit:** frame number is found in the TLB instantly (hardware lookup, no RAM access). Only **one RAM access** is needed — for the actual data.
- **TLB Miss:** frame number not in TLB; must access the page table in RAM. **Two RAM accesses** total (page table + actual data). After resolving, the {page number, frame number} pair is stored in the TLB so the next access to the same page will be a hit.

*The TLB is essentially a cache for the page table. Just like CPU caches avoid going to RAM for data, the TLB avoids going to RAM for page table lookups.*

### TLB and Context Switching

When a context switch occurs, the TLB entries from the previous process are stale — they contain mappings for a different process. If the new process looks up a page number and gets an old entry from the previous process, it will be sent to the wrong physical frame — a serious error (wrong data, or another process's memory).

**Two solutions:**

**Solution 1 — Flush TLB on every context switch:**
- Clear all TLB entries whenever a context switch happens.
- Simple, but costly: every context switch makes the TLB cold (empty). The next process must refill the TLB one miss at a time — losing all the performance benefit of the TLB for a while.
- Since context switches are very frequent (time-sharing, I/O blocks, etc.), this flushing overhead can seriously hurt performance. *This is not an ideal solution.*

**Solution 2 — Address Space Identifier (ASID): the better approach**
- Each TLB entry stores an extra field: the **ASID** (Address Space Identifier) — essentially the process ID that owns this entry.
- On a TLB lookup: the hardware checks both the page number *and* the ASID. A match is only declared (TLB hit) if *both* match the current process's ASID.
- Different processes' mappings can coexist in the TLB simultaneously — no flushing needed on context switch.

```css
TLB with ASID:
┌─────────┬─────────────┬──────────────┐
│  ASID   │ Page Number │ Frame Number │
├─────────┼─────────────┼──────────────┤
│   P0    │     10      │     100      │  ← P0's entry
│   P1    │     10      │      45      │  ← P1's entry for same page number
└─────────┴─────────────┴──────────────┘
```

**Example:**
- P0 ran and stored: page 10 → frame 100 in TLB (ASID = P0).
- Context switch to P1.
- P1 looks up page 10 in TLB: finds an entry for page 10, but its ASID is P0, not P1 → **TLB miss** for P1.
- P1 goes to its own page table in RAM → finds page 10 → frame 45 → stores {ASID=P1, page 10, frame 45} in TLB.
- Next time P1 accesses page 10: ASID = P1 matches → **TLB hit** → frame 45 returned instantly.

*This way, both P0 and P1 can have entries for page number 10 in the TLB without conflict — because the ASID distinguishes them.*

## Internal Fragmentation in Paging

Paging eliminates external fragmentation completely, but it introduces **internal fragmentation**.

**Why?** Pages are fixed size. A process's size is rarely an exact multiple of the page size.

**Example:** process size = 15KB, page size = 2KB.

```css
15KB / 2KB:
  7 full pages = 14KB of data
  1 partial page: holds only 1KB of actual data, but occupies a full 2KB frame
```

```css
Last frame in RAM (2KB frame):
┌──────────────┐
│  1KB  (data) │  ← actual process data
├──────────────┤
│  1KB  (FREE) │  ← wasted — internal fragmentation
└──────────────┘
```

- That 1KB of wasted space inside the last frame is **internal fragmentation**.
- It cannot be used by any other process — it is "inside" the frame already assigned to this process.
- *This waste is unavoidable with fixed-size pages. It must be accepted as the cost of eliminating external fragmentation.*
- In the worst case, internal fragmentation = (page size − 1) bytes. For a 4KB page size, up to 4095 bytes can be wasted per process.

---

## Page Table is Created at Process Creation

- When a process is created, the OS creates its page table and stores it in RAM.
- The base address of this page table is stored in the process's **PCB**.
- When the process is scheduled, the PTBR is loaded with this base address from the PCB.
- When a context switch happens, the PTBR is updated to the new process's page table address (from the new process's PCB).

---

## Summary: Paging vs Contiguous Allocation

| Property | Contiguous Allocation | Paging |
|---|---|---|
| External fragmentation | Yes — a core problem | **No — completely eliminated** |
| Internal fragmentation | Yes (fixed partitioning) / No (dynamic) | Yes — last page of every process |
| Contiguous free RAM required | Yes | No — scattered frames work fine |
| Address translation mechanism | Relocation Register + Limit Register | Page table + MMU |
| Translation performance | Fast (one addition) | Slow without TLB (two RAM accesses) |
| Translation with TLB | — | Fast (one RAM access on hit) |
| Per-process data | Relocation + Limit values in PCB | Page table base address in PCB |

> *Paging is one of the most important topics for OS interviews — be prepared to explain the concept, the address translation steps with bit-level breakdown, why it is slow, and how TLB fixes it.*
